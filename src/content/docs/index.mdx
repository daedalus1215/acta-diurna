## Philosophy

A few guiding principles behind these docs:

- Optimize for **future readers who are tired**
- Prefer **clear defaults** over clever solutions
- Separate **patterns** from **implementations**
- Write docs that survive **team and company changes**
- Treat documentation as part of the system, not an afterthought

---

## Portability note

These docs intentionally avoid:
- company names
- proprietary details
- internal URLs
- hard-coded service identifiers

Company-specific details are best added as a **thin overlay** when adapting
these docs to a particular environment.
